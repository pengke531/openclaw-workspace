# 经验教训

> 用途：L1 教训索引层。只记录高复用、会直接影响后续决策的经验，不写流水账。
> 读取规则：复杂任务、排障、架构调整前优先读；命中后再追溯详细案例。

## 核心教训

### 1. 先止血，再提速
- 复杂任务如果没有治理、闸门、止损规则，执行能力越强，乱试成本越高。
- 先建立任务分级、执行闸门、失败分类，再谈多 Agent 自治。

### 2. 最小闭环优先
- 先做可运行、可验证、可回滚的最小闭环，再做扩展。
- 不要一开始就堆一整套重链路。

### 3. 不稳定接口不能升级为主链路
- 即使局部能通，只要稳定性差，就不能作为系统默认依赖。
- 需要先有稳定替代方案或明确降级策略。

### 4. 依赖先验证可安装，再设计集成
- 在 ARM Windows 这类环境里，很多方案不是逻辑不对，而是依赖根本装不稳。
- 设计前先验证关键依赖、关键接口、关键 I/O 是否可用。

### 5. 记忆不能只躺在 daily log 里
- 如果信息只在 `memory/YYYY-MM-DD.md`，后续很容易“记过但用不上”。
- 必须做索引层和摘要层，支持按需加载。

### 6. 复杂任务必须区分执行与验证
- 只执行不验证，等于把错误延后。
- 至少要形成“执行者 + 验证者”的最低配回路。
