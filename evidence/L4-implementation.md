# L4 原始证据层 — 实现方案

> 目的：为 L4 提供明确的存储结构、访问规则和使用场景。

---

## L4 的定位

L4 是整个分层体系的最后一层，用于：
- 审计场景
- 争议/分歧解决
- 重要决策的原始证据追溯
- 外部内容真实性核实

**原则**：不主动加载，只在需要时访问。

---

## 三类 L4 存储

### 1. 对话原始记录 — `evidence/conversations/`

**存储内容**：
- 重要决策讨论的原始对话摘要
- 用户明确确认的关键指令（如“确认删除 X"）
- 任务交接时的上下文快照

**文件命名**：
```
conversations/YYYY-MM-DD-[session-id].md
```

**写入时机**：
- 重要决策完成后
- 用户明确确认某件事后
- 任务交接时
- 争议/冲突解决后

**内容格式**：
```markdown
# 对话快照 — YYYY-MM-DD

## 任务
[任务描述]

## 关键决策
- [决策点 1]
- [决策点 2]

## 用户确认
> [用户原话或确认摘要]

## 结论
[最终结论]

## 参与者
- 彭科（用户）
- 嬴（AI）
```

---

### 2. 工具完整输出 — `evidence/tool-outputs/`

**存储内容**：
- 关键工具调用的完整输出（API 返回、命令结果等）
- 外部调研的原始内容摘要
- 搜索/抓取结果的重要片段

**文件命名**：
```
tool-outputs/YYYY-MM-DD-[tool-name]-[hash].md
```

**写入时机**：
- 外部 API 调用结果重大时
- 调研结果需要存档时
- 工具输出用于重要判断依据时

**内容格式**：
```markdown
# 工具输出 — YYYY-MM-DD

## 工具
[工具名称 + 版本/参数]

## 调用目的
[为什么要调用这个工具]

## 原始输出
[完整输出内容]

## 关键提取
[从输出中提取的关键结论]

## 用途
[这个输出用于什么决策/判断]
```

---

### 3. 外部抓取内容 — `evidence/external-fetches/`

**存储内容**：
- 网页抓取的重要内容
- 外部文档摘要
- 市场调研原始数据

**文件命名**：
```
external-fetches/YYYY-MM-DD-[source]-[topic].md
```

**写入时机**：
- 外部内容作为重要决策依据时
- 需要留存证据的场景
- 调研内容可能被引用时

**内容格式**：
```markdown
# 外部抓取 — YYYY-MM-DD

## 来源
[URL 或来源描述]

## 抓取目的
[为什么抓取这个内容]

## 原始摘要
[提取的关键内容]

## 真实性备注
[是否有验证、是否可靠]

## 用途
[这个内容用于什么决策/判断]
```

## L4 访问规则

### 何时访问 L4

1. **审计场景**：检查某个决策是如何达成的
2. **争议解决**：用户说"我记得不是这样"，需要核实
3. **追溯学习**：理解某个技术方案的原始背景
4. **证据呈现**：需要向用户展示决策依据时

### 如何访问 L4

```
需要 L4 时：
1. 确定需要哪类证据（对话/工具输出/外部内容）
2. 按日期或关键词搜索对应目录
3. 读取目标文件
4. 将证据呈现给用户或用于判断
```

### L4 不要做的事

- **不要在日常任务中主动加载 L4**
- **不要把 L4 当作第二层记忆**
- **不要在 L4 存储普通任务内容**

---

## L4 与其他层的关系

| 层级 | 内容 | 写入触发 | 访问频率 |
|------|------|---------|---------|
| L0 | 总索引 | 重大变化 | 每会话 |
| L1 | 领域索引 | 每周整理 | 按需 |
| L2 | 每日日志 | 每日结束 | 按需 |
| L3 | 详细文档 | 重要决策/项目 | 按需 |
| L4 | 原始证据 | 关键节点 | 仅审计时 |

**数据流向**：L4 的内容是 L1-L3 的"原材料"，但只有最重要的才会沉淀到 L4。
