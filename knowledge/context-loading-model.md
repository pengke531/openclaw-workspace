# 上下文分层加载模型 Context Loading Model

> 目的：定义每次任务启动时如何按层加载记忆，平衡上下文完整性与读取效率。

---

## 加载层级定义

### L0 — 总索引层
**文件**: `MEMORY.md`  
**内容**: 身份、核心配置、查表索引、长期待办  
**加载规则**: 每次会话第一个读取  

### L1 — 领域索引层
**文件**:
- `memory/关于彭科.md` — 沟通偏好、称呼、核心追求
- `memory/技术配置.md` — 技术环境、可用工具、约束条件
- `memory/项目经验.md` — 各项目状态、结论、下一步
- `memory/经验教训.md` — 高复用教训，防止重复踩坑

**加载规则**: 
- 命中领域关键词时加载对应文件
- 不确定时优先加载 `技术配置.md` 和 `项目经验.md`

### L2 — 主题摘要层
**文件**:
- `memory/YYYY-MM-DD.md` — 每日原始日志
- 按主题/项目的独立摘要文件

**加载规则**:
- 任务涉及具体项目时，加载该项目最近的 daily log
- 每日结束后将关键决策从 L2 提炼到 L1

### L3 — 详细事实层
**文件**:
- 设计文档、技术方案、会议记录
- `docs/plans/` 下的完整蓝图
- `knowledge/` 下的详细规则文件

**加载规则**:
- 复杂任务、排障、架构决策时加载
- 由 L1/L2 命中后主动下钻

### L4 — 原始证据层
**文件**:
- 原始对话记录
- 工具完整输出
- 外部抓取内容

**加载规则**:
- 审计、争议核对时访问
- 不主动加载，作为最终证据层

## 加载决策树

```
收到任务
  │
  ▼
读取 L0 (MEMORY.md)
  │
  ▼
任务涉及哪些领域？
  ├─ 沟通/偏好 ──────────────────→ 加载 L1: 关于彭科.md
  ├─ 技术/工具/环境 ──────────────→ 加载 L1: 技术配置.md
  ├─ 项目/研发 ───────────────────→ 加载 L1: 项目经验.md
  └─ 教训/反思 ───────────────────→ 加载 L1: 经验教训.md
  │
  ▼
L1 提供足够上下文了吗？
  ├─ YES → 停止，继续任务
  └─ NO  → 继续
  │
  ▼
任务与哪个具体项目/主题相关？
  ├─ 有关 ──→ 加载 L2: 对应 daily log 或项目摘要
  └─ 无关 ──→ 判断为 L2 也不需要
  │
  ▼
L2 提供足够上下文了吗？
  ├─ YES → 停止
  └─ NO  → 继续
  │
  ▼
是复杂任务/排障/架构决策吗？
  ├─ YES → 加载 L3: 相关设计文档/蓝图/详细规则
  └─ NO  → 停止在 L2
  │
  ▼
是审计/争议核对吗？
  ├─ YES → 可访问 L4: 原始记录/工具输出
  └─ NO  → 不访问 L4
```

## 温度分区策略

### 热区 (Hot) — ≤7 天
- 状态：当前项目、活跃任务
- 加载优先级：高
- 典型文件：L0 + L1 + 最近 7 天 L2

### 温区 (Warm) — 7-30 天
- 状态：最近完成的项目、暂时搁置的任务
- 加载优先级：中
- 典型文件：按需从 L2/L3 加载

### 冷区 (Cold) — >30 天
- 状态：归档内容、历史项目
- 加载优先级：低
- 典型文件：仅审计时从 L3/L4 加载

---

## 分层写入规则

### 写入时同步更新索引
- 每日在 L2 写 raw log
- 每周将 L2 关键结论提炼到 L1 对应文件
- 每月将 L1 重大变更同步到 L0

### 防止信息孤岛
- 决策类信息必须同时写入 L1 对应文件
- 不能只留在当日 log 里

### L0 永远只存索引
- L0 不直接存储详细信息
- 所有详细信息存在对应 L1-L4 文件，L0 只做引用

---

## 验收标准

- [ ] L0 是唯一每次必读的文件
- [ ] L1/L2/L3 按命中加载，不全量加载
- [ ] L4 仅在审计/争议时访问
- [ ] 热区文件（≤7天）保持在 L0-L2
- [ ] 每日关键决策能追溯到 L1 索引
- [ ] 简单任务不触发不必要的深层加载
