# 任务分级标准 Task Grading

> 目的：在开工前快速判断任务复杂度、风险、是否需要计划、是否需要子会话/ACP。

## L1：直接执行型

**特征**
- 单一步骤或极少步骤
- 风险低、影响范围清晰
- 不涉及删除/外发/安装/改配置
- 不需要跨文件大范围修改

**默认执行模式**
- 主会话直接完成
- 可边做边答
- 完成后做最小自检

**确认要求**
- 通常不需要额外确认
- 若用户意图不清，再补问一句

**是否需要子会话**
- 不需要

**例子**
- 读取单个文件并总结
- 回答模型/配置现状
- 小范围文本修改

---

## L2：轻量多步型

**特征**
- 需要 3-5 步
- 涉及少量文件或少量工具调用
- 风险可控，但需要先想清顺序
- 需要简单验证

**默认执行模式**
- 先写简版 3-5 步计划
- 主会话执行
- 完成后给出验证结果

**确认要求**
- 若涉及写入、覆盖、创建较多内容，可先说明再做
- 不确定边界时先确认范围

**是否需要子会话**
- 通常不需要
- 仅在任务明显会拉长、需要独立上下文时考虑

**例子**
- 新建一组知识文件
- 做一轮轻量调研并输出总结
- 为现有方案补规则和测试说明

---

## L3：复杂执行型

**特征**
- 多阶段、多文件、多工具协同
- 有明显失败成本
- 需要计划、执行、验证分离
- 可能需要独立上下文或并行工作

**默认执行模式**
- 先计划，再执行
- 默认拆成“执行 + 验证”两角色
- 优先考虑 ACP / 子会话 / 独立执行单元
- 过程保留 checkpoint

**确认要求**
- 开工前明确目标、成功标准、边界
- 高风险写操作需显式确认

**是否需要子会话**
- 推荐需要
- 尤其适用于编码、重构、复杂调研、批量处理

**例子**
- 将一套能力正式做成可复用 skill
- 大范围代码改造
- 带验证闭环的多文件实现任务

---

## L4：高风险/系统级任务

**特征**
- 影响系统配置、外部系统、公开发布、删除/迁移关键内容
- 失败成本高或不可逆
- 需要清晰审计与回滚思路

**默认执行模式**
- 必须先计划
- 必须设置闸门与止损点
- 必须有显式确认
- 能分阶段就不要一次性全做

**确认要求**
- 必须确认目标、范围、影响、回滚方案
- 未确认不得执行高风险动作

**是否需要子会话**
- 视任务而定，但通常需要严格隔离与验证

**例子**
- 改 OpenClaw 配置并重启
- 删除大量文件
- 代表用户对外发送/发布
- 正式开源发布与远端仓库推送

---

## 默认路由规则

- **L1**：直接做，做完即答
- **L2**：先列简计划，再执行，再验证
- **L3**：先计划，默认拆执行/验证，必要时 spawn/ACP
- **L4**：必须确认 + 闸门 + 回滚思路，禁止直接开干

## 快速判断问题

1. 这是一步就能完成，还是明显多阶段？
2. 失败成本是低、中、高？
3. 是否涉及删除、外发、安装、改配置？
4. 是否需要多个角色或独立上下文？
5. 是否需要验证结果才能算完成？

如果第 3 项为“是”，至少按 **L3/L4** 处理；如果第 4、5 项同时为“是”，通常至少按 **L3** 处理。
