# 执行闸门 Execution Gates v2

> 目的：让任务在开工前先收敛，避免直接进入无边界试错。

## 开工前五问

1. **目标是什么？**
   - 最终要交付什么，不要只写动作。
2. **成功标准是什么？**
   - 什么结果出现，才算真的完成。
3. **风险边界是什么？**
   - 哪些内容不能碰，哪些动作不能直接做。
4. **工具路径是什么？**
   - 准备用哪些工具/文件/接口完成，是否有备选路线。
5. **是否需要子 Agent / ACP？**
   - 这件事是主会话直接做，还是应该拆给独立执行单元。

如果五问里有两项答不清，先别开工，先收敛。

---

## 高风险动作确认清单

以下任一命中，都不能按普通任务直接做：

- 删除文件 / 批量覆盖 / 不可逆修改
- 对外发送消息、邮件、公开发布
- 安装/卸载依赖、修改系统环境
- 修改 OpenClaw 配置、重启服务
- 推送远端仓库、创建公开仓库、发布开源项目

执行前至少确认：
- 影响范围
- 是否可回滚
- 用户是否明确授权
- 是否已有验证/备份思路

## 四类动作的额外闸门

### 1. 删除
执行前必须明确：
- 删除对象是什么
- 是否只删目标范围
- 能否恢复
- 用户是否明确要求

### 2. 外发
执行前必须明确：
- 发给谁
- 发什么
- 发送身份是谁
- 是否已经过用户确认

### 3. 安装
执行前必须明确：
- 为什么必须装
- 是否已有 first-class tool / 替代方案
- 安装失败的影响是什么
- 是否需要用户批准

### 4. 改配置
执行前必须明确：
- 改哪一段配置
- 为什么要改
- 改后是否会重启/影响现有行为
- 是否已有最小验证方案

---

## 默认执行规则

- 没有成功标准，不开工
- 没有边界说明，不做高风险动作
- 没有工具路径，不要边试边赌
- 没有验证方式，不算完成
- 连续两轮失败，必须停下来换路线或复盘
