# Evaluator-Optimizer 验证回路

> 实现者与验证者的双角色闭环，确保每个产物都经过严格验收。

---

## 核心回路

```
   ┌─────────────────────────────────────────────────┐
   │                                                   │
   │   规划 ──→ 实现 ──→ 产物 ──→ 验证               │
   │                            │                      │
   │                            ↓                      │
   │                       ┌─────────┐                │
   │                       │ 通过？  │                │
   │                       └─────────┘                │
   │                      ↙            ↘              │
   │                   YES                  NO         │
   │                    ↓                   ↓          │
   │              继续下一步      明确原因 → 回炉     │
   │                                       ↓           │
   │                               修正 → 重新验证    │
   │                                       ↓           │
   │                              三次不通过 → 人工确认│
   │                                                   │
   └─────────────────────────────────────────────────┘
```

---

## 验证者职责

### 必须做
1. **逐项对照**验收标准检查产物
2. **运行**实际测试（代码必须实际运行，不是只看）
3. **记录**每项验证的实际结果
4. **给出**明确结论：通过 / 不通过 + 原因

### 禁止做
1. 不跳过任何验收标准
2. 不自己动手改代码
3. 不给模糊结论（"差不多就行"不允许）
4. 不降低验收标准来迁就实现者

---

## 验证流程

### Step 1: 接收产物
- 确认产物格式符合 Handoff 要求
- 确认所有文件/模块已交付
- 有缺失立即要求补齐再验证

### Step 2: 逐项检查
- 对照 Handoff 的每条成功标准
- 记录每项的实际结果
- 区分"完全通过"/"部分通过"/"不通过"

### Step 3: 运行测试
- 代码必须实际运行
- 文档必须实际阅读
- 测试结果必须截图/记录

### Step 4: 给出结论

**通过**:
```
验证结果: 通过
综合评价: [简要评价]
耗时: [实际验证时长]
```

**不通过**:
```
验证结果: 不通过
不通过项:
  1. [标准项] — 不通过原因：[具体原因]
  2. [标准项] — 不通过原因：[具体原因]

修正建议:
  1. [针对每条不通过的修正建议]

验证耗时: [时长]
```

### Step 5: 决定下一步
- 通过 → 继续或交付
- 不通过（首次） → 回炉给实现者
- 不通过（二次） → 实现者修正后重新验证
- 不通过（三次） → 触发人工确认（主会话或用户）

---

## 验证标准分级

### P0 — 必须通过
- 功能正确性
- 数据安全性
- 破坏性操作的确认

### P1 — 应该通过
- 性能要求
- 兼容性
- 文档完整性

### P2 — 最好通过
- 代码质量
- 测试覆盖率
- 可维护性

**规则**: P0 必须全过；P1 建议全过；P2 视时间决定

---

## 特殊场景处理

### 场景 1: 实现者同时是验证者
**规则**: 禁止自己验证自己
**解决**: 找另一个角色或主会话验证

### 场景 2: 验证标准本身有问题
**规则**: 如果发现验收标准不合理，停止验证，记录问题
**解决**: 回到规划 Agent 重新定义标准

### 场景 3: 三次不通过
**规则**: 同一验收标准连续三次不通过
**解决**: 触发人工确认，可能是目标需要调整

### 场景 4: 用户催促跳过验证
**规则**: 禁止以时间压力跳过验证
**解决**: 说明风险，如果用户坚持，记录并上报

## 与 Post-Task Review 的关系

验证回路的通过不等于 Post-Task Review 的通过：

- **验证回路**: 验证产物是否满足验收标准（技术维度）
- **Post-Task Review**: 复盘整个任务执行过程（流程维度）

一个任务可能通过验证回路但在 Post-Task Review 中得低分（例如：实现过程敷衍但结果恰好满足标准）
